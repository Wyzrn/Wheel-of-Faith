# AI Character Image Generation — Implementation Plan

Status: DRAFT — awaiting user approval before execution
Last updated: 2026-05-31

## Goal

Every character generated by the 23-spin loop gets one AI portrait that:
1. Renders in the character card.
2. Renders in battle (replaces the letter sigil for that character).
3. Click → zoom to centered full-screen modal. Click backdrop → back to normal.

## North-star UX (zero-friction)

- Generation kicks off automatically right after spin 23 resolves, in parallel with the rest of the reveal — by the time the user finishes reading the summary, the image is usually already there.
- Card shows a skeleton with the existing sigil as visible fallback while loading. Never blocks card interaction.
- If generation fails, the sigil stays. No error toast — silent fallback. Optional manual "Regenerate portrait" button on the character page (logged-in only).
- One portrait per character, immutable. We do not regenerate on every page load.

## Architecture choices

### Model: Flux Schnell via fal.ai

| Why | |
|---|---|
| Cost | ~$0.003/image — 1,000 characters = $3 |
| Latency | ~1.5s typical, well under the reveal animation budget |
| Quality | More than enough for a 200–400px portrait; the description is already a strong prompt |
| Moderation | Permissive enough to handle the chaotic powers/weaknesses content (DALL-E + Gemini will refuse routinely) |
| Style lock | Supports seed parameter for consistency-per-character |

**Alternatives kept on the bench:** Replicate (same model, similar pricing, slightly slower cold-start); BFL Flux Pro direct (3–5× cost, marginal quality gain at this resolution); SDXL self-hosted (eliminates per-image cost but adds GPU ops — not worth it under ~10k images/month).

### Storage: Cloudflare R2

| Why | |
|---|---|
| Cost | ~$0.015/GB/mo. 100k portraits at 150KB each ≈ 15GB ≈ $0.22/mo. |
| Egress | Free to public — perfect for a static portrait CDN |
| API | S3-compatible — swap to S3 later with one config change |
| Public URL | One R2 custom domain → `https://portraits.wheeloffate.app/<shareId>.webp` |

**No existing object-storage integration in the project today** ([server/src/models/Character.ts](server/src/models/Character.ts) only stores metadata). This adds:
- 1 npm dep: `@aws-sdk/client-s3`
- 4 env vars: `R2_ACCOUNT_ID`, `R2_ACCESS_KEY_ID`, `R2_SECRET_ACCESS_KEY`, `R2_BUCKET`
- 1 R2 bucket + public binding (one-time Cloudflare dashboard task by you)

### Prompt strategy

Reuse the existing description pipeline at [src/lib/characterSummary.ts](src/lib/characterSummary.ts) (the same one that builds the card's "auto description"). Augment with a **locked style preamble** so every portrait reads as one set:

```
{styleLock}: digital painting portrait, dark fantasy comic-book splash page,
bold ink outlines, dynamic three-quarter pose, dramatic rim lighting,
centered subject on a moody gradient background, painted texture.

{character}: a {race} {archetype} named {name}.
Wields: {topPower}. {summary.text}.

Negative: photorealism, anime cel-shading, text/letters, watermark,
multiple subjects, extra limbs, low contrast.
```

Use the character's `shareId` as the deterministic seed so re-generation produces the same image.

## Backend additions

### 1. New module: `server/src/services/portraits.ts`
- `generatePortrait({ shareId, prompt, seed })` → uploads webp to R2, returns the public URL.
- Calls fal.ai Flux Schnell, converts response to webp at 512×512, uploads to `portraits/<shareId>.webp`.
- Idempotent on `shareId` — second call with same id just returns the existing URL.

### 2. New route: `POST /api/characters/:shareId/portrait`
- Auth: optional (anonymous saves should still get portraits — gate by per-IP rate limit instead).
- Rate limit: 5 / minute / IP (gen is cheap but a runaway loop would still hurt the wallet).
- Body: none. Server pulls the character, builds the prompt, calls `generatePortrait`, updates `Character.portraitUrl`.
- Response: `{ url }` on success, `{ error }` on failure (client treats failure as "stay on sigil").

### 3. Character model: add `portraitUrl: string | null`
- See [server/src/models/Character.ts](server/src/models/Character.ts:3). One new field, nullable.

## Frontend additions

### 5. Card integration — `src/components/CharacterCard.svelte`

### 6. Battle integration — `src/components/BattleArena.svelte`

### 7. New component: `src/components/PortraitZoom.svelte`

- Close: click backdrop or press Escape. Reverse animation back to origin.

## Open questions for you

1. **R2 vs S3 vs raw fal.ai CDN?** fal.ai returns hosted URLs that work for ~30 days — fine for an MVP but rough for permanence. My recommendation: R2 from day one (cheaper, permanent, costs ~$0.25/mo at any plausible scale). Alternatively, MVP-only: skip R2 and store the fal.ai URL directly, swap to R2 in phase 4. **Default: R2 from day one.**
2. **Backfill old characters?** ~N existing characters in the DB. Backfill = N × $0.003. Two options: (a) generate lazily on first view (free until viewed); (b) one-time batch backfill (predictable cost). **Default: lazy.**
3. **Regenerate button?** "I don't like this portrait" → one re-roll allowed per character, then locked. Free for logged-in owner, gated by 1/day per IP for anonymous. **Default: yes, owner-only, one re-roll.**
4. **Content moderation?** Flux Schnell will draw most things, but some weakness/power combos could produce content you don't want public. Cheapest mitigation: a blocklist of obvious problem terms in the prompt builder. More robust: post-gen NSFW classifier (adds ~$0.001 and ~300ms). **Default: blocklist now, classifier later if it becomes a problem.**
5. **Anonymous (logged-out) characters?** They get a `shareId` and exist in the DB. Do they get portraits too? Pros: viral sharing is prettier. Cons: every visitor's spin costs $0.003 even if they never come back. **Default: yes, but apply a stricter per-IP rate limit (3/hour) for anonymous spins.**
